# Phase 4.2: CI Operational Model — Implementation Plan

**Date**: 2026-04-13  
**Week**: 2 (Target completion: 2026-04-20)

## Objective

Extend CMDB to support Configuration Items (CIs) as first-class operational entities with:
- Lifecycle tracking (expiry dates, renewal reminders)
- Impact analysis (what depends on this CI)
- Relationship graph (BELONGS_TO, CONTAINS, DEPENDS_ON, etc.)

## Background

Current CMDB tracks **services** (containers, daemons) but not the underlying **configuration items**:
- SSL certificates (need expiry tracking)
- API keys/credentials (need rotation scheduling)
- DNS zones/records (need impact analysis before changes)
- Firewall rules (need dependency mapping)
- Backup jobs (need SLA tracking)
- Licenses/subscriptions (need renewal alerts)

Phase 4.2 makes CIs queryable with operational context.

## Deliverables

### 1. Database Schema ✅

**Option A: Extend `ops_cmdb` table** (preferred for simplicity)
```sql
ALTER TABLE ops_cmdb ADD COLUMN ci_type TEXT;
ALTER TABLE ops_cmdb ADD COLUMN expires_at TEXT;
ALTER TABLE ops_cmdb ADD COLUMN parent_ci TEXT;
ALTER TABLE ops_cmdb ADD COLUMN operational_status TEXT DEFAULT 'active';
```

**Option B: Create `ops_ci` table** (more normalized)
```sql
CREATE TABLE ops_ci (
    name TEXT PRIMARY KEY,
    ci_type TEXT NOT NULL,
    service_name TEXT REFERENCES ops_cmdb(name),
    expires_at TEXT,
    parent_ci TEXT REFERENCES ops_ci(name),
    operational_status TEXT DEFAULT 'active',
    metadata TEXT,
    created_at TEXT DEFAULT (datetime('now')),
    updated_at TEXT DEFAULT (datetime('now'))
);

CREATE INDEX idx_ci_type ON ops_ci(ci_type);
CREATE INDEX idx_ci_expires ON ops_ci(expires_at);
CREATE INDEX idx_ci_parent ON ops_ci(parent_ci);
```

**Decision**: Option B — CIs have different lifecycle than services, need separate table.

### 2. API Endpoints

#### `POST /ops/cmdb/ci` — Register CI
```json
{
  "name": "wildcard-cert-2026",
  "ci_type": "cert",
  "service_name": "caddy",
  "expires_at": "2026-10-15T00:00:00Z",
  "metadata": {
    "issuer": "Let's Encrypt",
    "domains": ["*.example.com", "*.example.org"]
  }
}
```

#### `GET /ops/cmdb/ci/{name}` — Get CI details
```json
{
  "name": "wildcard-cert-2026",
  "ci_type": "cert",
  "service_name": "caddy",
  "expires_at": "2026-10-15T00:00:00Z",
  "operational_status": "active",
  "days_until_expiry": 185,
  "metadata": {...},
  "relationships": {
    "depends_on": [],
    "used_by": ["caddy", "litellm"]
  }
}
```

#### `GET /ops/cmdb/ci/{name}/impact` — Impact analysis
```json
{
  "ci_name": "powerdns-api-key",
  "direct_dependents": ["powerdns", "caddy"],
  "indirect_dependents": ["all-services-using-dns"],
  "change_window_required": true,
  "risk_level": "high"
}
```

#### `GET /ops/cmdb/ci/expiring` — Expiry query
```json
{
  "expiring_in_30_days": [
    {"name": "slack-webhook-old", "ci_type": "credential", "expires_at": "...", "days_left": 15},
    {"name": "cert-staging", "ci_type": "cert", "expires_at": "...", "days_left": 7}
  ],
  "expiring_in_7_days": [...],
  "already_expired": [...]
}
```

### 3. Neo4j Graph Integration

**Schema additions**:
```cypher
// CI nodes
CREATE (ci:CI {name: "wildcard-cert-2026", ci_type: "cert", expires_at: "..."});

// Relationships
CREATE (caddy:Service {name: "caddy"})-[:USES]->(ci:CI {name: "wildcard-cert-2026"});
CREATE (cert-2026:CI)-[:RENEWS_TO]->(cert-2027:CI);
CREATE (zone-example-org:CI)-[:CONTAINS]->(record-api:CI);
```

**Population strategy**:
- On CI registration: Create CI node + relationships
- On service registration: Auto-detect CI dependencies from metadata
- Background sync: Periodically reconcile CMDB → Neo4j

## Risks + Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| Neo4j unavailable | Impact queries fail | Return partial results from SQLite |
| CI name collisions | Data corruption | Enforce UNIQUE constraint on name |
| Expiry timezone issues | Wrong alerts | Store all dates in UTC ISO8601 |
| Graph sync lag | Inconsistent view | Background sync every 5 minutes |

## Rollback Plan

If issues arise:
1. Revert database migration (DROP TABLE ops_ci)
2. Revert API router changes
3. CI data preserved in SQLite (can re-migrate later)
